# 火灾知识问答库问答系统：从 0 到 1 学习详解

## 1. 项目简要介绍

这是一个面向“火灾知识问答”场景的 RAG 系统。它把“文档管理 + 向量检索 + 大模型生成 + 会话管理”串成一条完整链路，目标是让用户上传文档后，能够基于知识库做可追溯、可持续对话的问答。

核心能力：
- 支持多格式文档上传（PDF / DOCX / TXT / MD）
- 文档自动解析、分块、向量化入库
- 支持向量检索、关键词检索、混合检索（RRF 融合）
- 支持普通聊天与知识库问答（SSE 流式返回）
- 支持用户认证、会话持久化、消息与引用来源保存
- 支持本地存储 / MinIO，支持 Chroma / Milvus

---

## 2. 技术栈与工程结构

### 2.1 技术栈
- 后端框架：Flask、Flask-CORS
- ORM 与数据库：SQLAlchemy + MySQL
- RAG 生态：LangChain
- 向量库：Chroma（默认）或 Milvus
- Embedding/LLM：HuggingFace Embedding，DeepSeek/OpenAI/Ollama
- 文本检索：BM25（rank-bm25 + jieba 中文分词）
- 依赖管理：uv（项目已配置清华源）

### 2.2 目录分层
- `app/blueprints`：路由层（HTTP 接口、参数接收、返回格式）
- `app/services`：业务层（知识库、文档处理、检索、问答、设置等）
- `app/models`：数据层（用户、知识库、文档、会话、消息、系统设置）
- `app/utils`：基础能力（数据库、日志、认证、文档加载、文本切分、LLM 工厂）
- `main.py`：应用启动入口

---

## 3. 一次请求是怎么跑通的（系统总流程）

你可以把项目看成 4 条主线：

1) 用户与权限线  
- 注册 / 登录后，`session["user_id"]` 维护登录态  
- 各接口通过装饰器判断是否已登录、是否拥有资源权限

2) 知识构建线（离线）  
- 上传文档 -> 存储文件 -> 记录文档元数据 -> 异步解析与分块 -> 向量化入库

3) 问答推理线（在线）  
- 用户提问 -> 检索相关分块 -> 拼接上下文 + Prompt -> LLM 流式生成 -> 返回答案与引用

4) 会话沉淀线（在线）  
- 用户问题和模型回答都落库，形成可回看、可追踪历史会话

## 4. 分层详解（按代码职责学习）

## 4.1 入口层（App 启动与初始化）
- 启动入口在 `main.py`
- `create_app()` 做了几件事：
  - 初始化数据库表
  - 注册蓝图（auth/knowledgebase/settings/document/chat）
  - 注入 `current_user` 到模板上下文
  - 开启 CORS

学习重点：
- Flask 应用工厂模式
- 蓝图拆分如何让项目从“单文件”升级到“分模块工程”

## 4.2 配置层（Config + .env）
- `app/config.py` 集中管理：
  - App 端口、调试开关、文件大小限制
  - MySQL 连接信息
  - 存储类型（local/minio）
  - 向量库类型（chroma/milvus）
  - LLM 与 API 参数

学习重点：
- 把“可变参数”从代码硬编码中抽离
- 用环境变量做“同一套代码，多环境运行”

## 4.3 数据层（模型设计）
- `User`：账号基础信息
- `Knowledgebase`：知识库主实体（名称、描述、封面、分块参数）
- `Document`：文档实体（文件路径、状态、分块数量、错误信息）
- `ChatSession`：会话头
- `ChatMessage`：会话消息体（角色、内容、sources）
- `Settings`：系统全局参数（检索模式、阈值、Prompt 等）

学习重点：
- 主实体 + 从实体建模
- `ondelete="CASCADE"` 的级联删除思想
- 元数据字段如何服务后续可观测性（status/chunk_count/error_message）

## 4.4 存储层（文件存储）
- 通过存储工厂屏蔽实现细节：
  - 本地存储：落盘到目录
  - MinIO：对象存储
- 上传时先落存储，再写数据库；失败时回滚清理，保证一致性

学习重点：
- 统一接口 + 多实现（策略模式思想）
- 失败补偿（上传失败清理已写入对象）

## 4.5 文档处理层（解析 -> 分块 -> 向量化）

### A. 上传阶段
- 接口接收文件，做扩展名和大小校验
- 文档元数据写入 `document` 表，状态初始为 `pending`

### B. 异步处理阶段
- 线程池提交异步任务
- 状态切到 `processing`
- 通过 `DocumentLoader` 按类型解析文本（PDF/DOCX/TXT/MD）
- 使用 `TextSplitter` 按知识库参数切分 chunk（chunk_size/chunk_overlap）
- 组装 LangChain `Document`，写入向量库
- 成功后状态 `completed`，并记录 chunk 数量
- 异常则状态 `failed`，记录错误信息

学习重点：
- 异步任务与主请求解耦
- 分块参数对召回率、上下文连贯性的影响
- 元数据设计（doc_id/chunk_id/chunk_index）如何支撑检索与引用

## 4.6 向量层（VectorDB 工厂）
- `VectorDBFactory` 根据配置返回 Chroma 或 Milvus 实例
- 业务层不用关心底层差异，只调用统一方法：
  - `add_documents`
  - `delete_documents`
  - `similarity_search_with_score`

学习重点：
- 面向接口编程
- 可替换基础设施（单机到分布式迁移）

## 4.7 检索层（三种模式）

### 1) 向量检索（semantic）
- 用向量相似度拿候选
- 距离转分数并阈值过滤

### 2) 关键词检索（lexical）
- 取集合文档，中文分词（jieba）
- BM25 计算相关性
- 分数归一化并阈值过滤

### 3) 混合检索（hybrid）
- 向量检索 + 关键词检索并行拿结果
- 使用 RRF 做结果融合
- 按融合分排序返回 top_k

学习重点：
- 语义检索与词面检索互补
- RRF 的核心思想：融合“名次”而非原始分数，鲁棒性更高

## 4.8 生成层（RAG）
- 检索得到相关 chunk 后，拼接 context
- 通过 `ChatPromptTemplate` 构造 system + human prompt
- 调用 LLM 流式生成
- 最后返回：
  - 回答文本
  - sources（命中分块、分数、文档名、内容片段）

学习重点：
- Prompt 模板化
- “检索 + 生成”职责边界
- 引用可追溯在业务中的价值（提升可信度）

## 4.9 接口层与实时输出（SSE）
- 问答接口使用 `text/event-stream` 连续推送 chunk
- 前端可边收边渲染，降低等待体感
- 响应结束发送 `[DONE]`

学习重点：
- SSE 与一次性 JSON 响应的差异
- 流式输出在 AI 应用中的交互优势

## 6. 面试维度知识点：可学习版（重点）

这一章不只是“列知识点”，而是把每个知识点拆成你在面试里真正会被追问的深度。你要达到的目标不是“记住名词”，而是能讲出：为什么这样设计、当时有哪些备选方案、你们做了什么取舍、上线后有什么结果。

### 6.1 后端工程能力（Flask + SQLAlchemy）

你至少要把“分层 + 鉴权 + 事务”讲顺。分层上，这个项目不是在路由里直接堆业务逻辑，而是蓝图只做参数接收和响应，服务层处理核心业务，模型层做数据表达。面试官听你讲分层时，真正想听的是你是否理解“可维护性来自职责边界”，而不是背“MVC”三个字。你可以明确说：如果把解析、分块、向量化都写在接口里，后续换存储、换向量库、加异步重试都会牵一发而动全身。

鉴权上你要讲清两层校验：第一层是“是否登录”，第二层是“是否有资源所有权”。很多同学只说登录态校验，这会被追问“那A用户能不能访问B用户的知识库”。你要强调 ownership 校验是防止水平越权的关键。事务上你要讲清“查询会话”和“写事务”分开，写操作必须确保原子性，比如文档状态切换、消息落库等都要么都成功要么都回滚。

你可以补一句工程价值：服务层封装事务，减少路由重复代码，并让异常处理策略统一；状态字段如 `pending/processing/completed/failed` 不是为了好看，而是为了前端可视化、重试控制和故障定位。

自测清单：
- 你能否在白板上画出路由层、服务层、模型层各自边界？
- 你能否解释“登录态合法但资源无权限”属于哪类安全问题？
- 你能否举出一个必须放在事务里的操作组合？

### 6.2 RAG 核心原理（从输入到答案）

这一小节是整个面试成败点。你要先讲“RAG解决什么问题”：纯大模型问答在垂直领域会出现知识过时、事实不可追溯、幻觉不可控。RAG通过“先检索领域文档，再让模型基于检索上下文作答”来把知识边界收紧。面试中不要只说“减少幻觉”，要补上“可追溯”——因为业务方更关心答案能否回指来源。

然后你要讲两条链路：离线链路负责把文档变成可检索知识（解析、分块、向量化），在线链路负责把用户问题转成可回答结果（检索、上下文组装、生成、回传引用）。离线/在线拆分的工程意义在于性能和稳定性：高耗时任务不阻塞用户请求，线上问答路径更短、延迟更可控。你还可以补一句：离线链路可重跑，在线链路要低延迟，这就是架构拆分依据。

上下文构建时要强调“不是把全文扔给模型”，而是把 top-k 命中的片段按结构拼接为 context，再通过 prompt约束模型仅基于 context 回答。这个“收敛输入范围”的动作是RAG质量核心。

自测清单：
- 你能否在 60 秒说完“RAG端到端时序”并包含离线/在线两条链？
- 你能否解释“检索命中率低”和“生成质量差”分别归属哪一层问题？
- 你能否说出为什么RAG在企业知识问答中比纯LLM更可落地？

### 6.3 文档处理与分块策略（高频追问点）

分块是被问到概率最高的技术点。你要能讲出：Embedding 模型有输入窗口限制，整篇文档向量化会导致语义平均化，检索时难以命中细粒度事实。分块后，每个片段都是一个可检索单元，召回粒度更细，引用也更明确。面试官如果继续追问“那为什么不越小越好”，你要回答：过小会造成语义断裂、上下文不足，虽然召回命中看起来高，但答案完整性下降。

`chunk_size` 和 `chunk_overlap` 是典型 trade-off。你可以这样讲：chunk_size 决定语义密度与检索颗粒度，overlap 决定跨段信息连续性。overlap 太小会丢边界信息，太大则产生重复片段，增加向量库冗余和检索噪声。好的调参不是拍脑袋，而是“基于错例迭代”：抽取代表性问题集，观察召回片段是否完整覆盖答案证据，再调参数。

元数据是另一个易被忽略但很加分的点。`doc_id/chunk_id/chunk_index/doc_name` 不仅用于显示引用，还用于删除重建索引、定位错误分块、排查重复写入。你要强调：没有稳定chunk_id，向量更新和追溯会变得很混乱。

自测清单：
- 你能否给出一个“分块太小导致回答碎片化”的真实例子？
- 你能否描述一套最小可行的分块调优流程（问题集、指标、迭代）？
- 你能否解释为什么删除文档时必须按 doc_id 清理向量？

### 6.4 检索工程（向量 / BM25 / Hybrid-RRF）

你要把这节讲成“检索策略不是炫技，而是业务适配”。向量检索擅长语义相近表达，比如“可燃物管理”与“火灾载荷控制”语义接近但词面不同；BM25擅长精确词项命中，比如法规条款号、设备型号、专有名词。中文场景里，两者经常互补，所以只做单路检索会有盲区。

为什么要 Hybrid + RRF？核心原因是不同检索器分数分布不一致，直接线性加权不稳定。RRF基于“排名位置”融合，天然更鲁棒。你要会讲“鲁棒”具体指什么：即使某次向量分数整体偏低，只要它排名靠前，仍能通过融合保留下来。这个解释通常会让面试官觉得你不仅知道概念，还理解工程可用性。

阈值和 top_k 关系要讲细：top_k 决定候选池大小，阈值决定噪声过滤强度。top_k 太小会漏证据，太大则上下文污染、token成本增加、生成更发散。阈值太高会空召回，太低会引入弱相关片段。你可以补充调参顺序：先适度放大 top_k 保召回，再通过阈值压噪声，最后看答案质量与延迟平衡点。

自测清单：
- 你能否说出“向量召回高但答案仍差”的两个可能原因？
- 你能否解释“混合检索不一定总是优于单路”的场景？
- 你能否讲出你会如何做离线检索评测（至少3个指标）？

### 6.5 Prompt 与生成控制（减少幻觉）

这一节你要讲“Prompt不是玄学，而是输入规范”。在项目里，system prompt定义角色与边界，例如“仅基于提供文档回答，不确定时明确说明”；human prompt注入具体 `context + question`。你要强调模板化带来的价值：可迭代、可版本化、可A/B测试，而不是每次手工改字符串。

参数控制同样会被追问。temperature 高意味着输出多样性更强，但事实任务中会增加不稳定性；max_tokens 影响回答长度和成本，太小会截断，太大可能啰嗦并增加幻觉暴露面。建议你面试时给出“任务导向参数策略”：问答场景通常采用较低temperature保障稳定，复杂解释类问题再适度上调。

“无答案策略”是防露馅关键。很多候选人会说“模型没命中就自由发挥”，这是错误信号。你要明确：检索为空或证据不足时，应该让模型直接说明“文档中暂无依据”，并提示可补充资料。这类策略体现你有产品责任感，而不仅是追求“看起来会答”。

自测清单：
- 你能否口述一段“拒绝编造”的 prompt 约束？
- 你能否解释“高温度+大top_k”为何可能放大幻觉？
- 你能否说明如何做 prompt 迭代评估（错例复盘维度）？

### 6.6 SSE 流式交互与会话持久化

你要把SSE讲成“交互体验优化 + 实现复杂度可控”的选择。LLM天然是增量生成，用 SSE 可以边生成边展示，显著降低用户等待焦虑。与WebSocket相比，SSE是单向推送，协议更轻，后端改造成本更低，特别适合“服务端持续输出、客户端被动接收”的问答场景。

但面试官可能会追问稳定性。你要讲到两个关键细节：第一，流式过程中异常也要按事件块返回，避免前端无响应卡死；第二，回答落库时机要谨慎。该项目选择“生成完整后再落库 assistant 消息”，这是为了避免半截内容进入历史会话，导致后续多轮对话上下文污染。这个点非常加分，因为它体现你考虑了数据一致性与用户体验统一。

你还能补一句会话持久化价值：不是只为“历史记录”，还用于审计与质量分析。sources落库后可以复盘“为什么给出这个答案”，为后续调参、错例分析提供依据。

自测清单：
- 你能否解释SSE断连后前端应如何处理重连与提示？
- 你能否说出“先落库再生成完”和“生成完再落库”的风险差异？
- 你能否给出一条流式错误处理的设计原则？

### 6.7 系统设计与工程化（扩展能力）

这一节是区分“会写功能”和“有工程视角”的关键。你要讲清工厂模式的价值：业务代码依赖抽象接口而不是具体实现，才能做到“同一套业务逻辑，不同基础设施切换”。例如开发环境用Chroma快速验证，生产环境切Milvus提升扩展性；LLM供应商也可按配置切换，不需要重写问答链路。

然后讲演进路径：当前文档处理是线程池异步，适合小规模；当并发与任务量上升，应该迁移到任务队列（如Celery）并引入重试、死信、监控。你需要给出“为什么”：线程池在多实例部署下不易统一调度，任务可观测性弱，失败恢复能力有限。这个回答能显示你理解系统从“能跑”到“可运营”的差异。

安全方面要主动提：密钥不应硬编码、日志要脱敏、权限校验要分层。监控方面至少要有处理耗时、检索命中率、问答失败率。这些指标不是锦上添花，而是后续优化的依据。

自测清单：
- 你能否说出当前架构扩容后的第一个瓶颈最可能在哪？
- 你能否给出“开发到生产”的最小迁移清单？
- 你能否解释为什么监控指标必须先于优化动作？

## 7. 面试官高频问题：深度版（含答题方向）

建议答题结构统一使用：**背景 -> 方案 -> 取舍 -> 结果**。  
这样更像“工程师回答”，不是背概念。

### 7.1 架构类问题

Q1：你们为什么要做 RAG，而不是直接调用大模型？  
答题方向：
- 背景：领域知识私有、更新频繁、准确性要求高。  
- 方案：离线构建索引 + 在线检索增强生成。  
- 取舍：增加了索引维护成本，但显著降低幻觉。  
- 结果：答案可追溯（sources），业务可信度更高。

Q2：你这个项目的端到端链路是什么？  
答题方向：
- 离线：上传 -> 解析 -> 分块 -> 向量化。  
- 在线：提问 -> 检索 -> 组装 prompt -> 流式生成 -> 落库。  
- 强调状态机与异常处理。

Q3：为什么离线与在线要拆开？  
答题方向：
- 离线任务耗时长且可重试；在线链路要低延迟。  
- 拆分后系统吞吐和稳定性更好，故障域也更清晰。

### 7.2 检索策略类问题

Q4：为什么支持向量、BM25、Hybrid 三种检索？  
答题方向：
- 向量擅长语义，BM25 擅长关键词精确匹配。  
- 混合检索弥补单路缺陷，尤其中文专业名词场景效果更稳。

Q5：RRF 是什么，为什么不用简单加权分数？  
答题方向：
- 各检索器分数分布不一致，直接加权不稳定。  
- RRF 以排名融合，对尺度不敏感，工程上更鲁棒。

Q6：top_k 和阈值怎么调？  
答题方向：
- 先保证召回，再压噪声。  
- top_k 过小会漏召回，过大会引入噪声与 token 成本。  
- 阈值要结合问题类型、文档质量、模型表现做实验调参。

### 7.3 文档处理类问题

Q7：chunk_size / overlap 怎么选？  
答题方向：
- 按文档类型与问题粒度分场景选。  
- 一般先中等 chunk + 小 overlap 起步，再根据错例做网格调参。  
- 指标看召回命中、答案相关性、响应时延。

Q8：文档处理失败如何保证一致性？  
答题方向：
- 文件上传与数据库写入有补偿逻辑。  
- 异步处理失败写 `failed + error_message`，支持重试/重处理。  
- 已处理文档重跑时先清旧向量再入新向量。

### 7.4 生成与交互类问题

Q9：你们怎么减少模型幻觉？  
答题方向：
- 通过检索增强限定信息来源。  
- Prompt 约束“仅基于文档回答，找不到就明确说明”。  
- 回传 sources 供用户核验。

Q10：为什么使用 SSE 而不是普通 JSON 响应？  
答题方向：
- 模型生成天然是增量输出，SSE 更契合。  
- 显著降低首字等待时间，体验更好。  
- 实现成本低于 WebSocket，适合单向推流场景。

Q11：会话消息为什么要持久化？  
答题方向：
- 支持上下文连续对话、问题复盘、产品可观测。  
- sources 一并落库，可用于审计与质量分析。

### 7.5 工程化与扩展类问题

Q12：为什么用工厂模式？  
答题方向：
- 解耦调用方与具体实现，降低替换成本。  
- 支持多供应商（LLM、向量库、存储）按配置切换。  
- 方便测试与后续扩展。

Q13：如果你继续优化这个系统，优先级怎么排？  
答题方向：
- 稳定性：线程池 -> 任务队列，提升可靠性。  
- 质量：引入重排序模型 + 评测集闭环。  
- 性能：缓存热点 query、批量写向量。  
- 安全：密钥治理、日志脱敏、权限边界收紧。

## 8. 面试实战答题模板（可直接套用）

这一章给你“能背、能改、能应对追问”的模板。你不用一字不差背，但一定要把结构记住。真正防止面试露馅的关键不是“台词华丽”，而是你每句话都有对应项目事实。

### 8.1 模板A：解释技术方案（最常用）

建议结构：**问题背景 -> 目标约束 -> 方案设计 -> 取舍原因 -> 实际结果**。

可直接套用的口语版：
- 问题背景：我们当时遇到的是______，比如纯模型回答不稳定、无法追溯来源。  
- 目标约束：目标是______，同时要满足______（如响应速度、可维护性、可扩展）。  
- 方案设计：我把链路拆成离线和在线，离线做______，在线做______。  
- 取舍原因：相比______方案，我们选现在这套是因为______，代价是______。  
- 实际结果：最终在______指标上更稳定，比如命中质量、响应体验、可追溯性都有提升。

如何防止露馅：
- 不要只说“用了RAG”，要说“RAG在你项目里具体做了哪几步”。
- 不要只说“效果变好”，要说“好在哪里、通过什么现象判断”。
- 如果没有精确数据，使用“明显降低/显著改善 + 具体场景例子”比乱报百分比更安全。

### 8.2 模板B：解释参数调优（高频追问）

建议结构：**初始设定 -> 问题现象 -> 调参路径 -> 验证方法 -> 定版结论**。

可直接套用的口语版：
- 初始设定：初始我们把 `chunk_size/top_k/threshold` 设为______。  
- 问题现象：上线后出现______，比如召回不全、答案离题、响应偏慢。  
- 调参路径：我们先调______保证召回，再调______过滤噪声，最后平衡______。  
- 验证方法：用______问题集做对比，观察______（命中片段、答案相关性、时延）。  
- 定版结论：最后确定______参数区间，并保留______作为场景化调整入口。

如何防止露馅：
- 面试官最怕听到“我感觉这样比较好”。你要强调“基于错例迭代”。
- 至少准备一个真实错例：比如术语问答BM25更稳、泛化问法向量更稳。
- 解释参数时要讲副作用：任何调优都不是单向收益。

### 8.3 模板C：解释故障排查（最能拉开差距）

建议结构：**现象 -> 缩小范围 -> 根因定位 -> 修复方案 -> 防复发机制**。

可直接套用的口语版：
- 现象：某阶段出现______，例如文档状态长期 processing、会话缺失回答。  
- 缩小范围：我先按链路拆分检查：上传/解析/向量写入/生成/落库，定位到______。  
- 根因定位：通过日志和数据库状态发现______，本质原因是______。  
- 修复方案：我调整了______，并补了______保护逻辑。  
- 防复发机制：新增了______监控或告警，并加了______回归用例。

如何防止露馅：
- 不要说“突然就好了”，一定要讲“证据链”。
- 尽量提到可验证信息：状态字段、日志关键字、上下游对账。
- 收尾必须有“预防动作”，体现工程闭环能力。

### 8.4 模板D：解释技术选型（容易被临时追问）

建议结构：**场景需求 -> 备选方案 -> 选型标准 -> 最终选择 -> 未来演进**。

可直接套用的口语版：
- 场景需求：我们需要______，重点关注______。  
- 备选方案：当时评估了______和______。  
- 选型标准：核心看______（开发效率、扩展性、运维复杂度、成本）。  
- 最终选择：先用______满足快速落地，再预留______实现平滑迁移。  
- 未来演进：随着规模上来，会切到______并补齐______能力。

如何防止露馅：
- 选型答案不要绝对化，强调“阶段最优”而不是“永远最优”。
- 你要能说出至少一个没选方案的缺点，证明你确实比较过。

### 8.5 模板E：解释你个人贡献（防止“像背团队稿”）

建议结构：**我负责的边界 -> 我做的关键动作 -> 我推动的结果 -> 我复盘的改进**。

可直接套用的口语版：
- 我负责的边界是______模块，接口/服务/数据层主要由我实现。  
- 关键动作是______，例如把检索模式做成配置化切换、补上会话与来源落库。  
- 带来的结果是______，比如排查效率提升、体验更流畅、迭代更快。  
- 复盘后我认为还可优化______，下一步计划是______。

如何防止露馅：
- 说“我负责”时要配细节，不然很容易被追问穿。
- 说“我们团队”时要说明你具体做了哪一段，不要全篇“参与了”。

### 8.6 30秒、60秒、3分钟答题压缩版

30秒版（电梯陈述）：
- 这是一个面向火灾知识问答的RAG系统，我主要负责后端链路：文档解析分块向量化、检索融合、流式问答与来源追溯。系统把离线构建和在线问答分层，保证了稳定性与响应体验。

60秒版（一轮追问）：
- 我们离线把文档做解析、分块、向量化，在线根据问题做向量/BM25/混合检索，再把命中片段拼接到prompt里流式生成。为了避免幻觉，回答必须基于检索证据，并把sources落库。工程上通过工厂模式解耦LLM、向量库、存储，便于从开发环境扩展到生产。

3分钟版（项目讲解）：
- 先讲业务痛点、再讲双链路架构、再讲检索策略与参数调优、最后讲一次故障排查和优化路线。你要把“设计原因 + 取舍 + 结果”说完整，而不是按模块背代码。

### 8.7 面试防露馅清单（临场前必看）

- 每个“我做了”的点，准备一个代码级细节或流程细节支撑。  
- 每个“效果提升”，准备一个业务现象证明，不要只说“提升很多”。  
- 每个“技术术语”，准备一句通俗解释，防止被要求白话复述。  
- 每个“优化建议”，准备为什么现在没做（阶段性取舍）和下一步计划。  
- 遇到不会的问题，不要硬编，按“我目前理解 + 我会如何验证”回答更稳。

## 9. 进一步优化建议（进阶）

- 安全：把默认密钥从代码中移出，统一走环境变量与密钥管理
- 稳定性：文档处理由线程池迁移到任务队列（如 Celery）
- 性能：对高频检索做缓存，批量向量写入，减少冷启动
- 质量：补齐服务层测试与集成测试，建立回归基线
- 观测：增加处理耗时、召回命中率、问答成功率等指标
